Programming Guide

Connect a Device

Before operating a device to implement image acquisition, parameter configuration, and other operations, you should first connect the device (open device).

Steps:

  1. (Optional) Call MV_CC_EnumDevices() to enumerate all devices corresponding to specified transport protocol on the subnet. The information of found devices is returned in the structure MV_CC_DEVICE_INFO_LIST by pstDevList.
  2. (Optional) Call MV_CC_IsDeviceAccessible() to check if the specified device is accessible before opening it.
  3. Call MV_CC_CreateHandle() to create a device handle.
  4. Call MV_CC_OpenDevice() to open the device.
  5. (Optional) Perform one or both of the following operations:
    • Get device information: call MV_CC_GetDeviceInfo() .
    • Get the optimal packet size (only for GigE cameras): call MV_CC_GetOptimalPacketSize() .
  6. (Optional)Perform other operations, such as image acquisition, image display, and parameter configuration. See the API chapter for details.
  7. Call MV_CC_CloseDevice() to close the device.
  8. Call MV_CC_DestroyHandle() to destroy the handle and release resources.


Sample Code:
(for reference only)
#include "MvCameraControl.h"
void main()
{
int nRet = -1;
void* m_handle = NULL;
//Enumerate all devices corresponding to specified transport protocol on the subnet.
unsigned int nTLayerType = MV_GIGE_DEVICE | MV_USB_DEVICE;
MV_CC_DEVICE_INFO_LIST m_stDevList = {0};
int nRet = MV_CC_EnumDevices(nTLayerType, &m_stDevList);
if (MV_OK != nRet)
{
printf("error: EnumDevices fail [%x]\n", nRet);
return;
}
int i = 0;
if (m_stDevList.nDeviceNum == 0)
{
printf("no camera found!\n");
return;
}
//Select the first found online device, and create a device handle.
int nDeviceIndex = 0;
MV_CC_DEVICE_INFO m_stDevInfo = {0};
memcpy(&m_stDevInfo, m_stDevList.pDeviceInfo[nDeviceIndex], sizeof(MV_CC_DEVICE_INFO));
nRet = MV_CC_CreateHandle(&m_handle, &m_stDevInfo);
if (MV_OK != nRet)
{
printf("error: CreateHandle fail [%x]\n", nRet);
return;
}
//Connect the device.
unsigned int nAccessMode = MV_ACCESS_Exclusive;
unsigned short nSwitchoverKey = 0;
nRet = MV_CC_OpenDevice(m_handle, nAccessMode, nSwitchoverKey);
if (MV_OK != nRet)
{
printf("error: OpenDevice fail [%x]\n", nRet);
return;
}
//......Other operations.
//Close the device and release resources.
nRet = MV_CC_CloseDevice(m_handle);
if (MV_OK != nRet)
{
printf("error: CloseDevice fail [%x]\n", nRet);
return;
}
//Destroy the handle and release resources.
nRet = MV_CC_DestroyHandle(m_handle);
if (MV_OK != nRet)
{
printf("error: DestroyHandle fail [%x]\n", nRet);
return;
}
}



Get Images Directly

The MvCamera SDK provides an API for acquiring images directly. You can call the API to get the images after you start streaming, or adopt an asynchronous mode (thread or timer) to get the images. See sample codes in GrabImage.cpp and GrabImage_HighPerformance.cpp .

  • Two methods for acquiring images directly (cannot be used at the same time):
    Method 1: Call MV_CC_StartGrabbing() to start acquisition. You need to open a buffer, and then call MV_CC_GetOneFrameTimeout() repeatedly in the application layer to get frame data with specified pixel format. When getting frame data, the upper application program should control the calling frequency according to the frame rate.
    Method 2: Call MV_CC_StartGrabbing() to start acquisition, and call MV_CC_GetImageBuffer() in the application layer to get frame data with specified pixel format. Then call MV_CC_FreeImageBuffer() to release buffer. When getting frame data, the upper application program should control the calling frequency according to the frame rate.
  • Application Scenario of Getting Images Directly:
    Both methods of getting images directly need to call MV_CC_StartGrabbing() to start acquisition. When getting the frame data, the application program should control the calling frequency according to the frame rate.
    Both methods support setting timeout period. The SDK will wait until data is returned, which will make streaming more stable.
  • Differences of the two acquisition methods:
    a. MV_CC_GetImageBuffer() should be used with MV_CC_FreeImageBuffer() in pairs. The data pointer of pstFrame should be released by MV_CC_FreeImageBuffer().
    b. Compared with MV_CC_GetOneFrameTimeout(), MV_CC_GetImageBuffer() is more efficient, and its stream buffer is allocated by SDK, while the stream buffer of MV_CC_GetOneFrameTimeout() should allocated by the developer.
  • Note:
    a. The above two methods and the method of acquiring image in callback function cannot be used at the same time. Only one method can be used at a time.
    b. The pData returns an address pointer. It is recommended to copy the data of pData to create another thread.



Steps:

  1. (Optional) Call MV_CC_EnumDevices() to enumerate all devices corresponding to specified transport protocol within subnet.
    The information of found devices is returned in the structure MV_CC_DEVICE_INFO_LIST by pstDevList.
  2. (Optional) Call MV_CC_IsDeviceAccessible() to check if the specified device is accessible before opening it.
  3. Call MV_CC_CreateHandle() to create a device handle.
  4. Perform one or multiple of the following operations to get/set different types of parameters.
    Get/Set Camera Bool Node Value MV_CC_GetBoolValue() / MV_CC_SetBoolValue()
    Get/Set Camera Enum Node Value MV_CC_GetEnumValue() / MV_CC_SetEnumValue()
    Get/Set Camera Float Node Value MV_CC_GetFloatValue() / MV_CC_SetFloatValue()
    Get/Set Camera Int Node Value MV_CC_GetIntValueEx() / MV_CC_SetIntValueEx()
    Get/Set Camera String Node Value MV_CC_GetStringValue() / MV_CC_SetStringValue()
    Set Camera Command Node Value MV_CC_SetCommandValue()

    Note:
    • All the open attribute value can be found in Table of Camera Parameter Nodes. The table provides detailed description about name, data type, value range, access mode, and introduction of each node.
    For example:
    1. Setting and getting acquisition mode of single-frame, multi-frame and continuous acquisition:
      MV_CC_SetEnumValue (handle, "AcquisitionMode", value)//Set acquisition mode. The type is Enumeration.
      MV_CC_GetEnumValue(handle, "AcquisitionMode", &stEnumValue)//Get the acquisition mode.
    2. Setting trigger parameters:
      1) Set trigger mode via MV_CC_SetEnumValue(handle, " TriggerMode", value)
      2) If trigger mode is enabled, then you need to configure trigger source via MV_CC_SetEnumValue(handle, " TriggerSource", value), including hardware trigger and software trigger.
      3) You can get and set the trigger delay time via MV_CC_GetFloatValue(handle, " TriggerDelay", &stFloatValue) and MV_CC_SetFloatValue(handle, " TriggerDelay", value).
      4) If software trigger is enabled, you need to call MV_CC_SetCommandValue(handle, "TriggerSoftware ") to trigger capture. For hardware trigger, input on the device local hardware will trigger capture.
    3. Setting image parameters:
      The image parameters listed below can all be configured via these general APIs: MV_CC_SetIntValueEx() , MV_CC_SetEnumValue() , MV_CC_SetEnumValue() , MV_CC_SetBoolValue() , MV_CC_SetStringValue() , and MV_CC_SetCommandValue()
      Image width and height, pixel format, frame rate, AIO offset, gain, exposure mode, exposure value, brightness, sharpness, saturation, grayscale, white balance, Gamma value, and other parameters.
      For other general APIs, see General Configuration APIs.
  5. Image Acquisition:
    1. Call MV_CC_StartGrabbing() to start streaming.
      • Call MV_CC_SetImageNodeNum() to set the number of image buffer nodes. When the number of acquired images is larger than this number, the earliest image data will be discarded automatically.
      • For raw image data, you can call MV_CC_ConvertPixelTypeEx() to convert the image pixel format, or call MV_CC_SaveImageEx3() to convert the image to JPEG or BMP format and save as a file.
    2. Call MV_CC_GetOneFrameTimeout() or MV_CC_GetImageBuffer() repeatedly in the application layer to get the frame data.
      • When getting frame data, the upper application program should control the calling frequency according to the frame rate.
  6. Call MV_CC_DisplayOneFrame() to input handle and start displaying frame.
  7. Call MV_CC_StopGrabbing() to stop acquisition.
  8. Call MV_CC_CloseDevice() to close the device.
  9. Call MV_CC_DestroyHandle() to destroy the handle and release resources.

Get Images in Callback Function

         The API MV_CC_RegisterImageCallBackEx() is provided for registering callback function. You can customize the callback function, and the obtained image will be automatically called back. This method can simplify the application logic. See sample code in Grab_ImageCallback.cpp .

  • Getting images in callback function:
    MV_CC_RegisterImageCallBackEx() can be used to register callback function. The callback function can be customized for calling back the acquired images. If you want to get RGB24 or BGR24 images, you can call MV_CC_RegisterImageCallBackForRGB() or MV_CC_RegisterImageCallBackForBGR() to register callback functions.
  • Notes:
    a. In callback functions, time-consuming operations and thread locks are not recommended, which may cause blocking.
    b. The pData returns an address pointer. It is recommended to copy the data of pData to create another thread.
